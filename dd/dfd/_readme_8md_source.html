11 ## What is `bit::memory`?
12 
13 This is an all-purpose memory management library. The primary emphasis on this library revolves around the memory allocators in the library,
14 designed to be as lightweight and efficient as possible. Custom C++ _logical_ concepts are designed and adhered to strictly within this library. Sadly these are not (yet) `C++20` or `C++ Concepts TS` concepts since they are not standardized.
15 
16 Additionally, this library offers utilities for the following concerns:
17 
18 - Standard Adapters: Adapter functionality between this library's `Allocator` concepts and the C++ standard library Allocator
19 - Endianness: Detects host machine endianness, and offers casting with `endian_cast` and `endian_swap` utilities
20 - Unaligned Access: Utilities for loading unaligned data in a standard-compliant manner
21 - Virtual Memory: Functionality for reserving and committing virtual memory pages in a cross-platform way (MacOS,Linux, and Windows)
22 - Aligned Memory: Functionality for allocating over-aligned memory
23 
24 This library defines a variety of `BlockAllocator`s and `Allocator`s -- and is extendable through user-defined custom types that
25 conform to the concepts defined here.
26 
27 ## Differences from Standard Allocators
28 
29 The `Allocator` concept in this library differs from the C++ standard's allocators in a few ways
30 
31 ### 1. Allocators are heterogenious
32 
33 The C++ Standard Library's `Allocator` concept defines every `Allocator` to be homogenous in its allocations; always allocating
34 a specific type `T`, and only converting to different allocators using `rebind`. This offers some level of optimizations, since
35 all allocations are of fixed size; but also can cause large amounts of code duplication due to the requirement of templates to
36 produce different instances of the same type, all based on type `T`.
37 
38 As a result, this library uses the SGI and HP approach of only allocating by `void*`, rather than by a given type. A wrapper
39 type called `std_allocator_wrapper` exists to wrap the given allocator in a manner that works with C++ standard containers.
40 
41 ### 2. Allocators support alignment
42 
43 Since the C++ Standard Library's `Allocator` only cares about the type `T` being allocated, there is no generalized support
44 for over-aligning types.
45 
46 The `Allocator` concept in this library _requires_ the presence of an alignment argument in order to work properly (although
47 this argument may be ignored in certain implementations).
48 
49 ### 3. Allocators require `try_allocate` rather than `allocate`
50 
51 This is a subtle difference. The C++ Standard Library's `allocate(...)` function is free to throw exceptions on failure; leaving
52 non-throwing allocators up to an implementation's discretion. The `Allocator` concept in this library requires `try_allocate`, which
53 may return a null pointer on failure and must be non-throwing.
54 
55 This allows for greater composition with other allocators, since an allocator that fails to allocate can be queried without requiring
56 `try`/`catch` statements (e.g. for fallback allocators)
57 
58 The optional `allocate` function _may_ throw. If this is not defined, then `allocator_traits<Allocator>::allocate(...)` defines a custom
59 one that -- on failure -- calls a global out-of-memory handler.
60 
61 ### 4. Allocator info
62 
63 The Allocator concept in this library optionally supports a member function named `infO()` that returns an `allocator_info` object.
64 This object is used for _naming_ the allocator so that it can be uniquely identified in case an error handler is called.
